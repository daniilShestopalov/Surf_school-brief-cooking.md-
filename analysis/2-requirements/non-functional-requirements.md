# Нефункциональные требования (Non-Functional Requirements)

| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| NFR-1 | Клиентское приложение реализуется как мобильное приложение (iOS/Android), интегрируемое с существующим REST API бэкенда. | Must have | `domain_model.md`, Введение |
| NFR-2 | Вся бизнес-логика (расчет рейтингов, статусов слотов) и атомарные проверки (защита от двойных бронирований) осуществляются на стороне бэкенда. Приложение выступает тонким клиентом. | Must have | `domain_model.md`, Раздел 3 (Бронирование) |
| NFR-3 | В качестве единственного канала информирования пользователей (напоминания, уведомления об отменах) используются Push-уведомления (без резервирования через SMS). | Must have | `requirements_summary.md`, Раздел 3 |
| NFR-4 | В MVP-версии приложения не реализуются элементы программы лояльности (бейджи, счетчики) и онлайн-эквайринг. | Won't have | `requirements_summary.md`, Раздел 2, 3 |
