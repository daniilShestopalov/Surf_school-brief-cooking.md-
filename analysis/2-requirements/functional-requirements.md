# Функциональные требования (Functional Requirements)

| ID | Формулировка | Приоритет | Источник |
| :--- | :--- | :--- | :--- |
| FR-1 | Система должна позволять пользователю авторизоваться по номеру телефона (с использованием SMS/OTP). | Must have | `requirements_summary.md`, Раздел 2 |
| FR-2 | При авторизации/регистрации система должна требовать явное согласие на обработку персональных данных. | Must have | `requirements_summary.md`, Раздел 3 |
| FR-3 | Система должна позволять заполнять информацию об аллергиях в профиле пользователя путем выбора из заранее заданного списка (тегов). | Must have | `requirements_summary.md`, Раздел 1 |
| FR-4 | Система должна отображать список доступных слотов (ClassProgram) на ближайшие 7 дней по умолчанию. Если слотов нет — показывать Empty State. | Must have | `domain_model.md`, Раздел 3 (Просмотр расписания) |
| FR-5 | Система должна предоставлять фильтр дат для поиска классов за пределами базовых 7 дней. | Should have | `domain_model.md`, Раздел 3 (Просмотр расписания) |
| FR-6 | Система должна позволять забронировать место на выбранный слот, создавая запись `Booking`. | Must have | `domain_model.md`, Раздел 3 (Бронирование) |
| FR-7 | При бронировании система должна позволять выбрать опцию аренды экипировки, если `available_equipment_stock` > 0. | Must have | `domain_model.md`, Раздел 3 |
| FR-8 | Если `available_equipment_stock` = 0 на выбранный слот, чекбокс аренды должен отображаться, но быть заблокированным с поясняющим текстом о том, что инвентарь уже забронирован. | Must have | `requirements_summary.md`, Раздел 1 |
| FR-9 | Сразу после успешного бронирования система должна отображать экран с инструкциями и реквизитами для перевода оплаты на карту. | Must have | `requirements_summary.md`, Раздел 3 |
| FR-10 | Система должна позволять клиенту отменить свое активное бронирование. | Must have | `domain_model.md`, Раздел 3 (Отмена) |
| FR-11 | При попытке отмены бронирования в день проведения класса, система должна выводить предупреждающее модальное окно о "сгорании" занятия, но с возможностью подтвердить отмену. | Must have | `requirements_summary.md`, Раздел 1 |
| FR-12 | В личном профиле клиента система должна отображать только активные (предстоящие) записи. Архив посещений не отображается (для MVP). | Must have | `requirements_summary.md`, Раздел 2 |
| FR-13 | После завершения класса система должна предложить клиенту оценить шефа от 1 до 5 звезд (без возможности написать текстовый отзыв). | Must have | `requirements_summary.md`, Раздел 2 |
| FR-14 | Карточка/возможность для выставления оценки шефу должна пропадать через 24 часа после завершения класса. | Must have | `requirements_summary.md`, Раздел 2 |
| FR-15 | Система должна отображать детальную информацию о выбранном слоте расписания (программа, описание, шеф, адрес, время, количество свободных мест). | Must have | `domain_model.md`, Раздел 3 (Бронирование) |
| FR-16 | При получении push-уведомления об отмене класса со стороны студии, система должна блокировать возможность повторного бронирования данного отмененного слота. | Must have | `domain_model.md`, Раздел 3 (Обработка форс-мажора) |
