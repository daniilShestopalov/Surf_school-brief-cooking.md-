# Технический отчет об онбординге: Кулинарная студия «Шеф-стол»

## 1. Общий архитектурный контекст (High-Level Overview)

- **Суть системы**: Клиентское мобильное приложение для записи на кулинарные мастер-классы. Приложение взаимодействует с существующим бэкендом, который выступает black-box источником истины для расписания, профилей шефов и управления транзакциями (атомарное бронирование).
- **Основные пользователи**: Только клиенты студии. Интерфейсы шефа и администратора/владельца находятся вне скоупа.
- **Границы MVP (In Scope)**:
  - Авторизация по номеру телефона (OTP).
  - Просмотр расписания слотов (по умолчанию 7 дней + фильтр по датам).
  - Бронирование места с опциональной арендой инвентаря.
  - Просмотр активных бронирований и управление отменами.
  - Оценка шефа (1–5 звезд) в течение 24 часов после завершения класса.
- **Вне рамок MVP (Out of Scope)**:
  - Онлайн-эквайринг (оплата реализована ручным переводом, с удержанием брони на 1 час).
  - Просмотр истории завершенных занятий (в профиле отображаются только активные записи).
  - Программы лояльности.
  - Внутренние механизмы админки (создание слотов, пересчет рейтингов шефов).

## 2. Ментальная модель данных (Data Architecture)

Архитектура данных строится вокруг нескольких ключевых сущностей, к которым приложение имеет ограниченный доступ (в основном Read-Only, кроме Client и Booking):

- **Client** (Read/Write): Клиент приложения. Ключевые поля: `phone` для авторизации [FR-10], `allergy_profile` — список тегов аллергенов в формате JSON / массива строк [FR-25].
- **ClassProgram** (Read-Only): Шаблон кулинарного класса. Поле `base_price` (Integer) хранит целочисленную базовую стоимость участия.
- **Chef** (Read-Only): Ведущий класса. Важное поле `rating` (Float), которое пересчитывается сервером на основе оценок клиентов.
- **Slot** (Read-Only): Конкретное проведение программы. 
  - Ресурсные поля: `max_capacity` и `available_seats` (Integer) контролируют доступность мест.
  - Аренда: `available_equipment_stock` и `equipment_tariff` (Integer) определяют остаток прокатного инвентаря и его стоимость [FR-65].
- **Booking** (Read/Write): Запись клиента на слот.
  - Содержит флаг аренды `needs_rental_equipment` (Boolean).
  - `expires_at` (Timestamp) — время сгорания неоплаченной брони [FR-80].
  - `status` (Enum) — состояния, через которые проходит бронь: `PENDING_PAYMENT`, `ACTIVE`, `CANCELLED_BY_CLIENT`, `CANCELLED_BY_STUDIO`, `COMPLETED`.

## 3. Техническая карта интеграции (API & Frontend Mapping)

Сквозная интеграция экранов UI с контрактом OpenAPI:

| Экран / Компонент UI | OpenAPI `operationId` | Метод | Эндпоинт |
| :--- | :--- | :--- | :--- |
| **SCR-001** (Авторизация: ввод номера) | `sendCode` | POST | `/auth/send-code` |
| **SCR-001** (Авторизация: ввод кода) | `verifyCode` | POST | `/auth/verify-code` |
| **SCR-003** (Главный: расписание) | `listSlots` | GET | `/slots` |
| **SCR-004** (Детали слота) | `getSlot` | GET | `/slots/{id}` |
| **BS-001** (Оформление бронирования) | `createBooking` | POST | `/bookings` |
| **SCR-006** (Профиль: активные записи) | `getBookings` | GET | `/bookings` |
| **BS-003** (Отмена бронирования) | `cancelBooking` | POST | `/bookings/{id}/cancel` |
| **BS-004** (Оценка шефа) | `rateChef` | POST | `/bookings/{id}/rating` |

**Логика передачи параметров при бронировании:**
1. При переходе с **SCR-003** (Расписание) на **SCR-004** (Детали слота) передается `id` слота.
2. При вызове **BS-001** (Шторка бронирования) этот `id` мапится в `slot_id`.
3. Внутри BS-001 формируется Payload для POST `/bookings`:
   - `slot_id` (передан из SCR-004).
   - `needs_rental_equipment` (высчитывается из состояния чекбокса аренды).
   - `seats_count` (в MVP жестко = 1).
4. После успешного создания (HTTP 201) из ответа извлекается ID брони и таймер `expires_at`, которые прокидываются на экран **SCR-005** (Реквизиты и таймер оплаты).

## 4. Критические и транзакционные UX/UI сценарии

### Процесс создания бронирования (BS-001 & UC-30)
1. **Валидация на клиенте:** При открытии BS-001 проверяется `available_equipment_stock`. Если значение > 0, чекбокс аренды активен. Если = 0, чекбокс отображается, но заблокирован с поясняющим текстом [FR-70].
2. **Калькуляция цены:** Итоговая сумма рассчитывается динамически на клиенте перед отправкой: `base_price` (слота) + `equipment_tariff` (если выбран чекбокс аренды) [FR-75].
3. **Бэкенд транзакция:** По запросу POST `/bookings` сервер стартует транзакцию с `SELECT ... FOR UPDATE`. Происходит проверка наличия `available_seats`. Если места есть, их счетчик декрементируется. Если `needs_rental_equipment = true`, декрементируется `available_equipment_stock`. Бронь переходит в `PENDING_PAYMENT`.

### Механизм идемпотентности
Для защиты от двойных списаний (в частности, повторного бронирования из-за слабого интернета), клиент обязан отправлять заголовок `Idempotency-Key` (UUIDv4) [BS-001_LOGIC].
- Ключ генерируется разово при инициализации сессии бронирования (или при первом нажатии).
- В случае транспортной ошибки сети (Timeout), когда клиент нажимает кнопку "Повторить", приложение обязано переиспользовать **тот же самый `Idempotency-Key`**, чтобы бэкенд не создал дубль, если первый запрос фактически дошел и был обработан сервером.

## 5. Стратегия обработки ошибок и Edge Cases

Управление ошибками базируется на жестком разделении бизнес-логики и проблем транспорта.

1. **Глобальная обработка `401 Unauthorized`:**
   - Если любой запрос возвращает 401, приложение производит принудительный разлогин, удаляет Bearer-токен из локального хранилища и перенаправляет пользователя на экран авторизации (SCR-001).

2. **Rate Limiting (`429 Too Many Requests`):**
   - Может возникнуть при запросе OTP (`sendCode`) или создании брони (`createBooking`). UI блокирует кнопку и показывает системное предупреждение ("Слишком много запросов, пожалуйста, подождите").

3. **Разделение транспортных и системных ошибок:**
   - Системный сбой `5xx`: Возникает при проблемах на бэкенде. Отображается системный алерт "Сервис временно недоступен. Попробуйте позже".
   - Сбой сети `NETWORK_ERR` (Timeout / Offline): Приложение не смогло достучаться до сервера. Показывается алерт "Отсутствует подключение к сети". Ретрай безопасен (используется идемпотентность).

4. **Специфические бизнес-ошибки при бронировании (Конкурентная гонка):**
   - **`409 Conflict` (SLOT_FULL):** Последнее место заняли за секунду до запроса клиента. UI: Модальное окно "Извините, места на этот класс только что закончились". Повтор невозможен [UC-30].
   - **`409 Conflict` (EQUIPMENT_OUT):** Места есть, но выбранный инвентарь закончился. UI: Снекбар "Инвентарь забронирован. Продолжить без аренды?". Пользователь должен явно снять чекбокс аренды и отправить запрос снова.
   - **`410 Gone`:** Слот был снят с расписания или отменен студией. UI: Ошибка "Слот больше недоступен для бронирования". Кнопка бронирования блокируется [FR-115].
