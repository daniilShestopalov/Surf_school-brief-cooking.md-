# Спецификация Mock-сервера и Инструкция для QA (Пасхалки)

Данный документ описывает поведение локального Mock-сервера, внедренного на этапе 5 (файл `MockApiHandler.kt`). Он позволяет мобильному клиенту работать автономно для проведения сквозного (E2E) тестирования бизнес-логики и UI.

---

## 1. Общие настройки сети
- **Задержка:** Для имитации реального сетевого взаимодействия на все моки установлена задержка (delay) в `500 мс`.
- **Подключение:** Клиентские запросы не уходят во внешнюю сеть, они перехватываются на уровне Koin с помощью `io.ktor.client.engine.mock.MockEngine`.

---

## 2. Спецификация API (Моки)

### 2.1 Авторизация (`/auth/send-code` и `/auth/verify-code`)
- **POST `/auth/send-code`**
  - **Запрос:** Любой валидный номер телефона.
  - **Ответ (200 OK):** Пустой JSON `{}`. Эмулирует успешную отправку SMS.
- **POST `/auth/verify-code`**
  - **Запрос:** Любой валидный код (например, `1111`).
  - **Ответ (200 OK):** Возвращается мок-токен для сессии:
    ```json
    {
      "accessToken": "mock-access-token",
      "refreshToken": "mock-refresh-token",
      "expiresIn": 3600
    }
    ```

### 2.2 Профиль и записи клиента
- **GET `/profile`**
  - **Ответ (200 OK):** Фейковые данные клиента "Иван Иванов" (с аллергиями).
- **PATCH `/profile`**
  - **Ответ (200 OK):** Пустой JSON `{}`. Симуляция успешного обновления данных профиля.
- **GET `/bookings` (Активные записи)**
  - **Ответ (200 OK):** Список бронирований. В моке жестко возвращается одна активная запись (статус `ACTIVE`) на мастер-класс (соотносится с ID первого слота).

### 2.3 Каталог расписания (`/slots` и `/slots/{id}`)
- **GET `/slots`**
  - **Ответ (200 OK):** Возвращает массив из 3-х карточек слотов с разными программами (Паста, Стейк Рибай, Борщ).
- **GET `/slots/{id}`**
  - **Ответ (200 OK):** Возвращает детализированную информацию о слоте (SlotDetails), включающую связанную программу (ClassProgram) и шефа (Chef), динамически подстраиваясь под ID запрошенного слота.

## 3. Инструкция по воспроизведению (Пасхалки)

Для проверки корректности отображения различных бизнес-состояний и ошибок при бронировании (POST `/bookings`), в моке запрограммированы специальные условия. QA-инженер может вызвать их, выбрав определенные мастер-классы в каталоге слотов.

### Пасхалка 1: Ошибка «Нет свободных мест» (409 SLOT_FULL)
1. **Шаг 1:** Откройте экран расписания.
2. **Шаг 2:** Выберите слот **"Стейк Рибай"** (внутренний ID: `slot-full-id`). В каталоге у него указано 0 свободных мест.
3. **Шаг 3:** Перейдите к бронированию и нажмите кнопку подтверждения.
4. **Результат:** Сервер перехватит запрос и вернет ошибку:
   ```json
   {
       "code": "SLOT_FULL",
       "message": "Нет свободных мест на данный слот"
   }
   ```
5. **Ожидаемое поведение UI:** Приложение должно обработать HTTP 409 Conflict и показать пользователю алерт/снекбар с соответствующим сообщением.

### Пасхалка 2: Ошибка «Нехватка инвентаря» (409 EQUIPMENT_OUT)
1. **Шаг 1:** Откройте экран расписания.
2. **Шаг 2:** Выберите слот **"Борщ"** (внутренний ID: `slot-no-equipment-id`). В каталоге у него указано 0 доступного инвентаря.
3. **Шаг 3:** При бронировании обязательно активируйте переключатель **«Нужна аренда инвентаря»** и нажмите кнопку подтверждения.
4. **Результат:** Сервер перехватит запрос и вернет ошибку:
   ```json
   {
       "code": "EQUIPMENT_OUT",
       "message": "Оборудование для данного слота закончилось"
   }
   ```
5. **Ожидаемое поведение UI:** Приложение должно обработать HTTP 409 Conflict и показать алерт/снекбар об отсутствии прокатного оборудования.

### Пасхалка 3: Успешное бронирование и Идемпотентность (200 OK)
1. **Шаг 1:** Откройте экран расписания.
2. **Шаг 2:** Выберите слот **"Итальянская паста"** (Обычный слот, ID: `e0b8a211-...`).
3. **Шаг 3:** Нажмите подтверждение бронирования.
4. **Результат:** Сервер вернет успешный ответ:
   ```json
   {
       "id": "b0b8a211-0000-0000-0000-book10000001",
       "status": "ACTIVE",
       "paymentUrl": null
   }
   ```
5. **Проверка `Idempotency-Key` (Тестирование дублей):**
   - Mock-сервер сохраняет заголовок `Idempotency-Key` (генерируется приложением) во временный `Set`.
   - Если QA-инженер (например, через Charles, перехватив запрос до Ktor Engine, или искусственно вызвав retry в коде) отправит повторный POST-запрос с тем же самым ключом `Idempotency-Key` на этот же слот, сервер не выдаст конфликт, а моментально отдаст **тот же самый успешный ответ**. 
   - Это гарантирует, что при обрывах сети пользователь не забронирует два места вместо одного.

### Пасхалка 4: Ошибка «Слот недоступен» (410 Gone)
1. **Шаг 1:** Откройте экран расписания.
2. **Шаг 2:** Выберите слот **"Удаленный класс"** (внутренний ID: `slot-gone-id`).
3. **Шаг 3:** Перейдите к бронированию и нажмите кнопку подтверждения.
4. **Результат:** Сервер перехватит запрос и вернет ошибку:
   ```json
   {
       "code": "SLOT_GONE",
       "message": "Данный слот больше недоступен"
   }
   ```
5. **Ожидаемое поведение UI:** Приложение должно обработать HTTP 410 Gone, закрыть экран бронирования и обновить список слотов (либо показать уведомление, что слот снят с публикации).

### Пасхалка 5: Ошибка «Слишком много запросов» (429 Too Many Requests)
1. **Шаг 1:** Откройте экран расписания.
2. **Шаг 2:** Выберите слот **"Популярный класс"** (внутренний ID: `slot-429-id`).
3. **Шаг 3:** Перейдите к бронированию и нажмите кнопку подтверждения.
4. **Результат:** Сервер перехватит запрос и вернет HTTP 429 с заголовком `Retry-After: 30` и телом:
   ```json
   {
       "code": "TOO_MANY_REQUESTS",
       "message": "Слишком много запросов. Повторите позже."
   }
   ```
5. **Ожидаемое поведение UI:** Сетевой перехватчик `HttpResponseValidator` должен выбросить `RateLimitException(30)`, а UI отобразить соответствующий снекбар о таймауте и необходимости подождать 30 секунд.
