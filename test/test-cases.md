# Набор тест-кейсов для мобильного приложения Surf School

Данный документ содержит подробные формальные тест-кейсы, покрывающие основной функционал приложения (Авторизация, Расписание, Бронирование, Профиль), включая позитивные и негативные сценарии, а также обработку краевых случаев.

---

## Авторизация

### TC-001: Успешный вход в приложение (Ввод OTP)
**Preconditions (Предусловия):**
* Приложение установлено, пользователь не авторизован.
* Сервер доступен, сеть работает исправно.

**Steps to reproduce (Шаги тестирования):**
1. Открыть приложение и перейти на экран входа.
2. Ввести корректный номер телефона.
3. Отметить чекбокс "Согласие на обработку персональных данных".
4. Нажать кнопку "Получить код".
5. Дождаться SMS с OTP-кодом и ввести полученный код на экране верификации.

**Expected result (Ожидаемый результат в UI):**
* Код успешно проверяется. 
* Происходит авторизация пользователя (если профиль новый, перенаправление на экран ввода имени, иначе — на главный экран).
* Пользователь видит главный экран (Расписание).

**Status:** [ ] Untested

---

### TC-002: Ошибка авторизации: Неверный OTP-код
**Preconditions (Предусловия):**
* Приложение открыто, выполнен запрос OTP-кода, открыт экран ввода кода.

**Steps to reproduce (Шаги тестирования):**
1. Ввести заведомо неверный OTP-код (например, "0000").
2. Дождаться ответа от сервера.

**Expected result (Ожидаемый результат в UI):**
* Поле ввода подсвечивается красным.
* Отображается сообщение "Неверный код" или аналогичное.
* Авторизация не происходит, пользователь остается на экране ввода кода.

**Status:** [ ] Untested

## Расписание и каталог

**Preconditions (Предусловия):**
* Пользователь авторизован.
* В расписании на ближайшие 7 дней есть доступные слоты.

**Steps to reproduce (Шаги тестирования):**
1. Открыть приложение (перейти на главный экран).
2. Дождаться загрузки списка слотов.

**Expected result (Ожидаемый результат в UI):**
* Отображается список слотов на ближайшие 7 дней.
* Для каждого слота видна базовая информация (название класса, время, количество свободных мест, стоимость).

**Status:** [ ] Untested

### TC-004: Отображение каталога слотов (Empty State / Нет слотов)
**Preconditions (Предусловия):**
* Пользователь авторизован.
* На бэкенде удалены/отсутствуют слоты на ближайшие 7 дней (или выбранный в фильтре период).

## Бронирование

### TC-005: Успешное бронирование слота (с добавлением аллергий и арендой инвентаря)
**Preconditions (Предусловия):**
* Пользователь авторизован.
* В профиле указаны теги аллергий (влияет на данные бронирования).
* Выбран слот с доступными местами (`available_seats` > 0) и доступным инвентарем (`available_equipment_stock` > 0).

**Steps to reproduce (Шаги тестирования):**
1. На главном экране выбрать слот и перейти на экран деталей.
2. Нажать кнопку "Забронировать".
3. На экране подтверждения отметить чекбокс "Аренда инвентаря" (убедиться, что он активен и стоимость пересчитана).
4. Нажать "Подтвердить запись".

**Expected result (Ожидаемый результат в UI):**
* Бронь успешно создается.
* Пользователь перенаправляется на экран оплаты (реквизиты) с таймером на 1 час (статус "Ожидает оплаты").
* Аллергии, указанные в профиле, автоматически прикреплены к брони (проверка через API/БД).

**Status:** [ ] Untested

---

### TC-006: Ошибка бронирования: Нехватка мест (409 SLOT_FULL)
**Preconditions (Предусловия):**
* Пользователь авторизован.
* Открыт экран подтверждения бронирования слота.
* *Эмуляция:* Пока пользователь находится на экране подтверждения, через API/админку последнее место в слоте бронируется другим пользователем (`available_seats` становится 0).

**Steps to reproduce (Шаги тестирования):**
1. Нажать кнопку "Подтвердить запись".

**Expected result (Ожидаемый результат в UI):**
* Бронирование не проходит (сервер возвращает статус 409 SLOT_FULL).
* Появляется модальное окно с текстом: "Извините, места на этот класс только что закончились" (или аналогичное).
* Пользователь не переходит на экран оплаты.

**Status:** [ ] Untested

---

### TC-007: Ошибка бронирования: Нехватка инвентаря (409 EQUIPMENT_OUT)
**Preconditions (Предусловия):**
* Пользователь авторизован.
* Открыт экран подтверждения бронирования, выбран чекбокс аренды инвентаря.
* *Эмуляция:* Пока пользователь находится на экране подтверждения, другой клиент забирает последний инвентарь (`available_equipment_stock` становится 0).

**Steps to reproduce (Шаги тестирования):**
1. Нажать кнопку "Подтвердить запись".

**Expected result (Ожидаемый результат в UI):**
* Сервер возвращает статус 409 EQUIPMENT_OUT.
* Появляется Snackbar (или модальное окно) с сообщением: "Инвентарь уже забронирован другим. Продолжить без аренды?".
* Предоставляется возможность повторить попытку бронирования после снятия чекбокса аренды.

**Status:** [ ] Untested

---

### TC-008: Тест идемпотентности (Повторная отправка запроса при обрыве сети)
**Preconditions (Предусловия):**
* Пользователь авторизован, находится на экране подтверждения бронирования слота с доступными местами.

**Steps to reproduce (Шаги тестирования):**
1. Нажать "Подтвердить запись".
2. Немедленно (с помощью прокси типа Charles/Proxyman) симулировать тайм-аут или обрыв сети на клиенте, **но при этом дать запросу уйти на сервер**.
3. Дождаться, пока UI приложения выдаст ошибку сети, оставив пользователя на экране подтверждения.
4. Восстановить сеть.
5. Нажать "Подтвердить запись" повторно.

**Expected result (Ожидаемый результат в UI):**
* При первом запросе бронь успешно создана на бэкенде.
* При повторном нажатии (с тем же Idempotency-Key/UUID в заголовках) сервер возвращает успешный ответ (существующую бронь), а не создает дубликат и не выдает ошибку о нехватке мест, если это было последнее место.
* Пользователь переходит на экран оплаты/реквизитов без дублирования записей.

**Status:** [ ] Untested

## Профиль и Активные записи

### TC-009: Просмотр активных записей в профиле и успешная отмена
**Preconditions (Предусловия):**
* Пользователь авторизован.
* У пользователя есть как минимум одна предстоящая (активная) запись.
* Отмена считается "Заблаговременной" (до 00:00 дня проведения класса).

**Steps to reproduce (Шаги тестирования):**
1. Открыть раздел "Профиль" ("Мои записи").
2. Убедиться, что активная запись отображается в списке.
3. Нажать на карточку активной записи.
4. Нажать кнопку "Отменить запись".
5. Подтвердить отмену в появившемся стандартном модальном окне ("Штраф не взимается").

**Expected result (Ожидаемый результат в UI):**
* Запись успешно отменяется.
* Карточка исчезает из списка "Мои записи" (или переходит в историю/отмененные).

**Status:** [ ] Untested

---

### TC-010: Поздняя отмена бронирования (Штраф)
**Preconditions (Предусловия):**
* Пользователь авторизован, есть предстоящая бронь.
* Отмена считается "Поздней" (день проведения класса).

**Steps to reproduce (Шаги тестирования):**
1. Открыть "Профиль", перейти в карточку активной записи.
2. Нажать "Отменить запись".

**Expected result (Ожидаемый результат в UI):**
* Появляется предупреждающее модальное окно с текстом: "К сожалению, продукты уже закуплены, оплата сгорит. Вы всё равно хотите отменить запись?" (согласно FR-90).
* После подтверждения запись отменяется на сервере, карточка скрывается из списка активных.

**Status:** [ ] Untested
