# Кулинарная студия «Шеф-стол» (Учебный проект)

Проект представляет собой разработку клиентского мобильного приложения для записи на кулинарные мастер-классы. Разработка ведется на основе реального брифа от заказчика.

## Описание брифа

**Заказчик:** Артём, владелец кулинарной студии «Шеф-стол».
**Цель:** Создать удобное мобильное приложение для клиентов, чтобы автоматизировать процесс записи на классы (сейчас всё ведется вручную в WhatsApp и Google-таблицах).

**Основной функционал (MVP):**
- Просмотр расписания кулинарных классов (по умолчанию на ближайшие 7 дней).
- Запись на класс с возможностью выбора дополнительных опций (аренда фартука и ножей).
- Указание информации об аллергиях при записи.
- Отмена записи клиентом.
- Оценка шефа после прохождения класса.
- Получение push-уведомлений (напоминания, отмена класса студией).

**Технические ограничения и скоуп:**
- В рамках проекта разрабатывается **только клиентское мобильное приложение и API-контракт** для него.
- Вся серверная инфраструктура, админка и интерфейсы для шефов уже существуют (black-box).
- Защита от двойных бронирований и управление расписанием обеспечивается на стороне существующего бэкенда.

## Основные задачи проекта

В рамках работы над брифом были поставлены и выполнены следующие задачи:

1. Сгенерировать с ИИ требования к MVP (user stories + основные сценарии) — [Подробности в аналитике](./analysis/README.md).
2. Получить от ИИ архитектурный план и схему данных — [Архитектура](./development/architecture-blueprint.md), [Схема данных](./development/data-schema.md), [Доменные сущности](./development/domain-entities.md), [План реализации](./development/implementation-plan.md).
3. Реализовать хотя бы 3 фичи — [Реализация клиентского приложения](./development/client/README.md), [Лог задач](./development/tasks-log/README.md).
4. Сгенерировать тест-кейсы, найти и исправить от 1 до 3 багов — [Тест-кейсы](./test/test-cases.md), [Журнал багов](./bagfix-log/README.md).

*Для каждой задачи и бага заведен отдельный .md документ с описанием симптома/цели, требованиями, процессом реализации/доработки, прикрепленными промптами и результатами ручного тестирования.*

## Запуск проекта

Поскольку клиентское приложение реализовано на Kotlin Multiplatform с использованием Compose Desktop, его можно запустить локально:

```bash
cd development/client
./gradlew run
```

## Структура репозитория

Для удобной навигации проект разбит на следующие директории:
- `/analysis` — Требования, user stories и аналитика по брифу.
- `/development` — Исходный код клиентского приложения, архитектурные документы и логи реализации фич.
- `/test` — Сгенерированные тест-кейсы для проверки приложения.
- `/bagfix-log` — Журнал найденных багов и отчёты об их исправлении.

## Инструменты реализации

Проект был реализован с использованием искусственного интеллекта:
- **Модель:** Gemini 3.1 Pro (High)
- **AI-агент:** Antigravity (в качестве разработчика, аналитика и помощника)

## Итоги и выводы

Работа над проектом стала ценным опытом взаимодействия с AI-агентами в рамках полного цикла разработки — от системной аналитики до написания кода и тестирования.

**Ключевые инсайты:**
- **Перспективы AI-разработки:** Использование ИИ-агентов демонстрирует огромный потенциал в ускорении рутинных задач, генерации базового кода и проектировании архитектуры.
- **Проблема «чёрного ящика» при vibe-coding:** Процесс выявил реальную сложность отслеживания логики AI на глубоких уровнях. Делегирование больших блоков логики ИИ без пошагового контроля иногда приводит к неочевидным багам, требующим ручной отладки.
- **Погружение в предметные области:** Практика в рамках проекта позволила прикоснуться к трём ключевым сферам IT (Аналитика, Разработка, QA) и лучше понять их взаимосвязь.

**Результат:**
На данный момент мы получили частично рабочий MVP клиентского приложения. Базовые сценарии покрыты, интерфейс функционирует и связывается с Mock-сервером, а найденные в процессе тестирования критические баги задокументированы и устранены. 

При наличии дополнительного времени на детальное погружение, точечный рефакторинг и доработку архитектуры, проект можно было бы довести до стадии полноценного production-ready продукта. Тем не менее, как учебный кейс для изучения возможностей ИИ-кодинга, этот проект полностью выполнил свою задачу.
